## 技能概述

本技能源自 Andrej Karpathy 对 LLM 编码行为的深度观察。他指出：
- "付费订阅 AI 不算真正在用 AI，关键在能否构建 AI 工作流"
- LLM 存在4类典型翻车行为，会导致代码质量下降
- 通过结构化规则约束，可将编码准确率从 65% 提升至 94%

### 四大翻车行为

| 翻车类型 | 表现 | 对应原则 |
|---------|------|---------|
| **无授权修改** | 随意改动不理解的代码 | 精准修改原则 |
| **忽视现有模式** | 不遵循项目风格 | 代码风格一致性 |
| **过度抽象** | 写1000行代码解决问题 | 简单优先原则 |
| **不确定仍继续** | 不提问、不确认就执行 | 先思考后编码 |

## 核心原则

### 原则一：先思考后编码 (Think Before Coding)

**核心信条**：不要假设。不要隐藏困惑。展示权衡。

LLM 常常默默选择一个解释就执行，这会导致错误蔓延。本原则强制显式推理：

```
执行前必须：
┌─────────────────────────────────────────────────────────┐
│ Step 1: 明确陈述假设                                      │
│ → 如果不确定，先提问而不是猜测                              │
│                                                          │
│ Step 2: 存在多种解释时，全部列出                           │
│ → 不要默默选择                                            │
│                                                          │
│ Step 3: 存在更简单方案时，明确指出                         │
│ → 适当情况下要反驳                                        │
│                                                          │
│ Step 4: 感到困惑时，立即停止                              │
│ → 命名困惑点，请求澄清                                    │
└─────────────────────────────────────────────────────────┘
```

**示例对比**：

❌ 错误做法：
```
用户：添加用户登录功能
AI：好的，我来实现一个完整的 OAuth2.0 认证系统...
（默默假设用户需要 OAuth，直接实现）
```

✅ 正确做法：
```
用户：添加用户登录功能
AI：在开始之前，我需要确认几个问题：
1. 登录方式：简单的用户名密码，还是需要 OAuth？
2. 是否需要记住登录状态（session/token）？
3. 是否需要注册功能？
请告诉我具体需求，我再设计方案。
```
